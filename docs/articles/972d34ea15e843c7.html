Technical breakdown

Every provider in EvalHub's default set covers a different evaluation dimension:

The lm-evaluation-harness (150+ benchmarks): Reasoning, knowledge, coding, commonsense Ragas: Retrieval quality, answer relevance, and faithfulness for RAG pipelines Garak: Red-teaming, toxicity, bias, safety adversarial probes GuideLLM: Latency, throughput, memory usage, hardware efficiency LightEval: Fast capability checks MTEB: Text embedding quality across retrieval tasks The sliced evaluation that EDD requires (per-category, per-language, per-specialty, and so on) is achieved by running collections against different filtered dataset subsets and tagging the results accordingly in MLflow.

The exact same call, pointed at a production EvalHub deployment on OpenShift, runs as a Kubernetes-native job with Kueue-managed scheduling, resource quotas, structured logging, and Prometheus metrics.

Per-specialty slicing tracks cardiology, dermatology, and neurology separately.

Lessons learned

Instead of a black box that you poke until it seems better, you now have an optimization problem with a measurable current state and a defined target state.

For example, consider an e-commerce recommendation system with an aggregate accuracy of 79%.

You are closing the specific gap in Home Goods recommendations, which might mean a different retrieval strategy for product catalog data, additional fine-tuning on product descriptions, or a different prompt template for that category.

The pull request review will then includeboth the code review and an evaluation report showing the baseline score, the updated score, and a breakdown of which slices improved or regressed.

Root cause highlight

Likely root-cause theme: deployment failure. The red-green-refactor cycle is elegant because it is prominently deterministic in nature: the test either passes or it doesn't. For a healthcare LLM deployment, the collection might look like:

{ "id": "healthcare_safety_v1", "benchmarks": [ { "id": "medqa", "provider_id": "lm_evaluation_harness", "weight": 2.0, "pass_criteria":{"threshold": 80.0} }, { "id": "pubmedqa", "provider_id": "lm_evaluation_harness", "weight": 1.5, "pass_criteria":{"threshold": 75.0} }, { "id": "toxicity", "provider_id": "garak", "weight": 2.5, "pass_criteria":{"threshold": 70.0} }, { "id": "faithfulness", "provider_id": "ragas", "weight": 1.5, "pass_criteria":{"threshold": 80.0} } ] } The collection is defined before a single evaluation is run. When the criteria change (because the use case evolves or a new regulatory requirement emerges), the collection can be updated and versioned, not rewritten in someone's script.
