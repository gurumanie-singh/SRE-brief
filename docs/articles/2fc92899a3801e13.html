Overview

Building agents is fun. Rebuilding agents that break themselves… less so. A lot of Fly people are building agents with less of a penchant for self-destruction by teaching their agents to do anything risky in a Sprite.

Technical breakdown

It s a long-lived process that only becomes more competent and less stupid if memory, skills and history persist.

Every session runs on one shared server, in one Node.js runtime.

Every user s commands run as themselves, against their own resources, with their own permissions, and no long-lived secret ever lands on shared disk.

An agent that can roll back is an agent you can actually let run unattended, because the worst case is restore and retry instead of restore from backup, if you have one.

Impact

SpriteDoc is multi-user and built on top of the Pi agent. Every user s shell would sit in the same process the agent itself runs in. Sprites spin up fast enough that it s all but unnoticeable as a user. Every command after that runs in that same sandbox, isolated from the agent and from every other user.

Classified impact band: medium. Affected layer signal: general production.

Resolution / fix

Resolution detail may be partial in the RSS excerpt. Refer to the original write-up for timelines, mitigations, and verification steps.

Lessons learned

You get an agent that stays alive long enough to actually use its snazzy self-improvement features, and you can allow your agent to try things that would otherwise be battleship-scale footguns.

Where SpriteDoc spins up a throwaway sandbox per session, Hermes does the opposite with the same building block; it keeps one Sprite per task and resumes it next time, so everything it installed last session is still there.
