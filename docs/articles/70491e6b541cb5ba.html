Overview

In early 2023, Slack faced a foundational challenge: serving Large Language Models (LLMs) at enterprise scale with the security, reliability, and performance our customers expect. Over three years, we evolved from basic infrastructure to orchestrating a sophisticated multi-cloud architecture. We didn’t just want shiny new models; we needed a system resilient to regional outages and GPU scarcity.

Technical breakdown

This required our teams to manage the operational lifecycle, including cross-region IAM roles, balanced routing across model endpoints, proactive capacity planning, and auto-scaling logic.

We used feature flags to slowly bleed traffic over, so if anything looked even slightly off, we could yank it back in seconds.

We kept high-volume, latency-sensitive features on dedicated capacity (Provisioned Throughput) to ensure a consistent “snappy” feel.

To maximize uptime for a global user base, we deployed these containers across multiple AWS regions.

Resolution / fix

Rollout: Implemented gradual traffic shifts via feature flags and instant rollback capabilities, ensuring 100% availability during the switch.

From the customer’s perspective, the experience remained seamless; they continued to receive high-quality results without ever knowing a complex failover had occurred behind the scenes.

First, no matter how many failovers we engineered within a single cloud, we remained susceptible to any potential provider-wide outage.

As Slack AI scaled to millions of users, we realized that true enterprise-grade reliability and a best-of-breed model strategy required looking beyond any single provider.

Lessons learned

We faced three primary taxes:

Scaling Latency: Initialization times prevented instantaneous scaling.

Because Bedrock serves as the primary launchpad for new LLMs, we were able to deliver model upgrades and quality improvements to users weeks or months earlier than was possible on SageMaker, directly enhancing the user experience across the entire Slack AI suite.

This granular optimization led to immediate performance gains: ~10% improvement in quality metrics for complex reasoning tasks.

~67% reduction in latency for high-velocity, low-token workloads.

If one model or provider slows down, the system instantly reroutes the request to a better-performing alternative, making the underlying complexity completely invisible to the user while maintaining a seamless experience.

More importantly, it removed the technical bottleneck we faced in Phase 2: because we were no longer locked into multi-month commitments, we regained the freedom to migrate features to different models. To bridge the gap, we engineered a Spillover Pattern: if a sudden surge pushed us beyond our reserved limits, excess requests automatically “spilled over” to on-demand endpoints, ensuring we never dropped a request due to capacity ceilings.
