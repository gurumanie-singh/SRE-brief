Overview

Confidential virtual machines (CVMs) are becoming increasingly ubiquitous. It is possible to provision an AMD SEV-SNP or an Intel TDX CVM from major cloud providers in a matter of minutes. QEMU/KVM support for the technologies has also significantly matured, making it possible to deploy an on-premise CVM setup.

Technical breakdown

The attacker does not have access to the VM’s memory and/or CPU register state because such attacks are presumably mitigated by CVM technologies (e.g., AMD SEV-SNP and Intel TDX).

CVM technologies provide reliable guarantees that the key cannot be extracted from VMs memory.

QEMU/KVM support for the technologies has also significantly matured, making it possible to deploy an on-premise CVM setup.

The main focus of these technologies is to provide data confidentiality guarantees for the VM at runtime, but it is equally important to consider the confidentiality of data at rest (e.g., in the situation where the VM gets restarted) migrates to another host or simply when the VM is required to be stateful.

This section explores several hypothetical attack scenarios where an attacker with read/write access to the CVM's storage attempts to breach data confidentiality, integrity, or availability, despite the use of traditional Linux disk encryption tools like dm-crypt/LUKS.

Lessons learned

UKI’s initramfs must be hardened to prevent interactive access.

To prevent this attack, PCR12(system extensions)/PCR13(configuration extensions)/PCR14(MOK) need to be added to the set of the PCRs used for secret sealing.

Encrypted data (non-root) disks which are mounted from the interactive system must be sealed to the guest identity to prevent secret unlocking if the disk is connected to a different guest which is under the attacker's control.
