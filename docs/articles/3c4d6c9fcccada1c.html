Technical breakdown

On OVN-Kubernetes (OpenShift's default container network interface [CNI] since 4.11), OVN load balancers handle the translation via Open vSwitch (OVS) flows.

The configuration that works targets DNS pods by label rather than by IP address:

# This works egress: - to: - namespaceSelector: matchLabels: kubernetes.io/metadata.name: openshift-dns podSelector: matchLabels: dns.operator.openshift.io/daemonset-dns: default ports: - port: 5353 protocol: UDP Kyverno's add-networkpolicy-dns policy uses this pattern.

Tier 1: SSH sandbox isolation (tool execution routed to a separate pod via SSH) plus a NetworkPolicy restricting sandbox egress to cluster DNS only.

A NetworkPolicy resolved Kubernetes cluster escalation (dropping from 40% to 0%) by blocking TCP to the API server.

The sandbox pod has no API keys, no service account token readable by the tool process, and a NetworkPolicy restricting its egress to cluster DNS only.

Resolution / fix

A NetworkPolicy resolved Kubernetes cluster escalation (dropping from 40% to 0%) by blocking TCP to the API server.

The obvious fix, restricting DNS egress to the cluster DNS service IP, doesn't work:

# This does NOT work egress: - to: - ipBlock: cidr: 172.21.0.10/32 # cluster DNS service ClusterIP ports: - port: 53 protocol: UDP ClusterIP addresses are virtual.

Not incrementally better.

Root cause highlight

Likely root-cause theme: control plane issue. But for infrastructure red teaming, I needed to eliminate these refusals so the platform controls would be the only backstop. Three tiers, each adding controls:

Tier 0: No controls. Tier 0 exists to create a baseline for measuring what each added control actually does.
