Overview

Security Insights provides actionable security recommendations for every Cloudflare account. To find these insights, we perform regular scans for all accounts, zones, and DNS records, looking for potential security risks and misconfigurations. However, two key issues emerged.

Technical breakdown

By looking closely at our code, SQL queries, logs, and metrics ( especially metrics!), we were able to increase our capacity without simply adding more pods or partitions.

Investigating our API timeouts We noticed several strange behaviours in our internal API as we tried to scale:

A large number of requests were triggering client-side timeouts

Many checkers were spending 20-90% of their processing time on a single API call

When triggering a large volume of scans, our throughput would start high and deteriorate

All of these problems had the same root cause: latency .

As a result of this latency, database queries from the Amsterdam API instance took much longer, holding connections from our client-side connection pool open.

The trade-offs are that we’d have more work to re-do if our process crashed midway through a batch, and our memory usage would be slightly increased.

With the large volume of requests that we were making to the API, the connection pool was quickly becoming exhausted, leading to timeouts waiting for a free connection.

Impact

In the worst case, these messages can take minutes or hours to process compared to the average case of seconds or milliseconds. Making sure that we’re finding these issues for all of our customers is crucial to our aim of building a better Internet for everyone. This is the story of how we increased scanning throughput for Security Insights by more than 10x, enabled security insights for millions of customers, and doubled our scanning frequency for all customers.
